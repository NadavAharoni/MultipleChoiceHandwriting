# Project Overview – MultipleChoiceHandwriting

## Purpose

This project automatically reads handwritten multiple-choice answer sheets and produces an Excel file containing the detected answers.

The implementation should be robust enough to process the 29 scanned exams.
Some manual intervention is ok.

On one hand, this is a one-time task, just to analyze the given 29 exams, since I need to grade them.

The reason for creating code and not asking a LLM to just do the work is half practical:
I asked Claude to convert the images to an Excel sheet. It went on and on and consumed so many tokens that my Pro/Teams session was consumed.

I have a theory that asking the LLM to write code would end up being cheaper in terms of tokens. 

There is also a side benefit: I gave a short talk recently, basically saying that in some cases asking LLMs to write code, is cheaper than asking them to solve the direct problems. Sometime LLMs do this themselves: they write a small script and execute it in their virtual machine.

So this task might be an example in which I will compare asking the LLM to write a script, with asking the LLM to do the full analysis.

# Overall Pipeline

Scan
→ Preprocessing
→ Table Detection
→ Answer Cell Segmentation
→ Handwriting Recognition
→ Answer Normalization
→ Excel Export

Each stage should have a single responsibility.

---

# Form Characteristics

This is **not** a general OCR project.

The answer sheets all use the same printed template.

Characteristics:

- Fixed layout.
- Two answer tables.
- Questions 1–10 on the right.
- Questions 11–20 on the left.
- Students write only Hebrew letters:
  - א
  - ב
  - ג
  - ד
  - ה
- Multiple answers are allowed (e.g. א,ג).

Exploit the fixed layout whenever it improves robustness.

---

# Design Principles

## Geometry First

Locate and normalize the answer table before attempting OCR.

After normalization, later stages should work with fixed coordinates.

## Robustness Over Generality

This project targets one specific exam form.

Prefer the most reliable solution over the most general one.

## Incremental Development

Each stage should be independently testable.

Continue generating debug images where useful.

---

# Handwriting Recognition Ideas

Recognition should not rely solely on isolated characters.

Useful techniques may include:

- Writer-specific adaptation.
- Clustering similar handwriting from the same student.
- Confidence propagation across answers.
- Using exam constraints to improve low-confidence predictions.

Example:

If a student's handwritten א is confidently recognized several times, uncertain samples can be compared against those examples.

Recognition should be performed before applying answer-key or exam-specific constraints.

---

# Engineering Philosophy

- Prefer deterministic processing.
- Keep stages independent.
- Validate every stage before continuing.
- Produce useful debug artifacts.
- Optimize for correctness and maintainability.
